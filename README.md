# 품:ON - 유기동물 보호소 수용현황 분석 & 연결 플랫폼

> 공공데이터 기반 보호소 과밀 예측 및 분산 보호 연결 서비스

## 프로젝트 소개

본 프로젝트는 공공 유기동물 데이터와 보호소 현황 데이터를 분석하여 보호소의 수용 압박 상태를 시각화하고, 향후 위탁 보호 연결 및 분산 보호를 지원하는 중개 플랫폼입니다.

단순 유기동물 조회 서비스가 아닌, **데이터 분석 기반 문제 해결형 서비스**라는 점이 핵심입니다.

## 기획 배경

- 매년 **10만 마리 이상**의 유기동물 발생
- 보호소별 수용 능력 편차 심각
- 과밀 보호소 발생 → 안락사 증가 → 구조 악순환
- 기존 서비스는 "공고/입양 정보 제공" 중심, **보호소 운영 관점 데이터 분석 서비스 부족**

**→ 보호소 수용 수용가능지수 시각화 + 분산 보호 연결 기반 서비스 기획**

## 핵심 기능

### 1. 보호소 수용 수용가능지수 분석
- 구조동물 수 / 보호소 최대 수용 가능 수 기반 현재 수용 비율 계산
- 수용가능지수 지수 시각화 (위험 70% 이상 / 경고 40% 이상 / 여유 39% 이하)

### 2. 지역별 유기동물 분포 대시보드
- Low Poly Map 기반 지역별 수용 현황 시각화
- Chart.js 기반 통계 차트
- 위기 보호소 TOP 5 실시간 표시

### 3. 날짜별 데이터 누적 저장
- Firebase RTDB에 일자별 데이터 저장
- API 호출 최소화를 위한 데이터 캐싱 구조

### 4. 사용자 인증 시스템
- 보호소 / 임시보호자 회원 구분
- 역할 기반 접근 제어

### 5. 향후 확장 기능 (기획 단계)
- 과밀 예측 시스템 (일일 증가량 기반)
- 위탁 보호 가능자 매칭 시스템
- 지자체 협업 운영 도구 확장

## 데이터 처리 구조

```
공공 유기동물 조회 API
        ↓
일자별 데이터 수집
        ↓
Firebase RTDB 저장
(rescuedAnimals/yyyy-mm/dd)
        ↓
보호소 전화번호 기준 매칭
        ↓
최대수용수 대비 현재 수용 비율 계산
        ↓
대시보드 시각화
```

## 데이터베이스 구조 설계 의도

| 설계 결정 | 의도 |
|----------|------|
| **RTDB 선택** | 실시간 동기화와 단순 구조에 적합, SDK 사용으로 CORS 문제 없음 |
| **날짜 기준 + 보호소 기준 이중 구조** | 원본 데이터 보존 + 분석용 구조 분리 목적 |
| **공공 API 호출용 프록시 서버 구축** | 공공데이터 API의 CORS 제한 우회 및 API 키 노출 방지 |

## 개인 회고

### 김선관 (임시보호자 페이지)
> **도전**: 공공 API와 Firebase 데이터를 병합하는 구조를 처음 설계하면서, 기획-개발 간 일정 조율에 어려움을 겪음

> **성장**: 데이터 병합 로직과 사용자 지역 기반 매칭 구조를 설계하며 **비동기 데이터 흐름 설계 역량**을 키움. 향후 프로젝트에서는 데이터 구조 설계를 기획 단계에서 확정하는 것이 중요함을 체득

### 심수완 (회원가입/로그인)
> **도전**: Firebase Auth 기반 인증 구조와 사용자 DB 분리 설계를 담당하면서, 실제 사용자 테스트까지 진행하지 못한 점이 아쉬움

> **성장**: 역할 기반 접근 제어 로직을 구현하며 **인증/인가 설계의 기본기**를 다짐. 사용자 피드백 기반 UX 개선의 중요성을 인식하고, 향후 프로젝트에서는 MVP 단계부터 사용자 테스트를 병행할 계획

### 서우주 (보호소 페이지)
> **도전**: 수용 가능 지수 기반 필터 구조 설계와 카카오맵 API 연동을 동시에 진행하면서, 우선순위가 낮은 기능 구현과 디자인 통합에 어려움을 겪음

> **성장**: 복잡한 필터 로직을 설계하고 외부 API를 연동하며 **조건 기반 데이터 필터링 설계 역량**을 키움. 디자인 시스템의 중요성을 체감하고, 컴포넌트 단위 설계의 필요성을 인식

### 김준민 (입양동물 페이지)
> **도전**: 프로그래밍 경험이 부족한 상태에서 Chart.js를 활용한 데이터 시각화와 위험보호소 우선순위 로직을 구현해야 했음

> **성장**: 처음 접하는 라이브러리를 학습하며 **차트 기반 데이터 시각화 구현 역량**을 키움. 부족한 부분은 팀원과 협업하며 해결했고, 기술 학습과 구현을 병행하는 방법을 터득

### 권창은 (메인 대시보드)
> **도전**: 메인 대시보드 UI/UX 설계와 Firebase RTDB 연동을 담당하면서, 상세화면 연동과 디자인 통일까지 완성하지 못한 점이 아쉬움

> **성장**: 대시보드 전체 레이아웃과 데이터 정렬 로직을 직접 설계하며 **프론트엔드 구조 설계 역량**을 키움. 초기 단계에서 디자인 시스템을 확립하는 것이 중요함을 체득

## 팀 회고

### KEEP(유지할점)
- 실제 사회 문제 기반 주제 선정 : 현실적인 사회 문제를 기술로 해결하려는 방향성, 데이터 기반 문제 해결 서비스
- 공공데이터 API 연동 경험 : Open API 활용, 데이터 정제+매칭 로직 구현 경험
- 시각화 중심 대시보드 구현 경험 : 각종 지표 데이터를 시각화한 서비스로 발전

### PROBLEM(문제점)
- 초기 설계 미흡 : 후반에 데이터 구조 수정이나 코드 리팩토링 부담이 큼
- Git 협업 관리 미숙 : 브랜치 전략, 커밋 컨벤션, PR 활용을 계획했으나 실행단계에서 원활하게 진행 못한 점
- 설정한 팀 컨벤션의 변동사항 조율과 적용이 부족해서 디자인 일관성 미흡

### TRY(시도할점)
- 기획 단계에서 데이터 구조와 흐름을 설계한 다음 구현 필요 : 데이터 구조도 기획단계에서 설계 마무리(필요한 데이터 정제과정 시간소요 고려)
- Git 협업 방식 개선 : 구현 전 브랜치 생성 > PR 기반 병합 > 배포 구조로 개발 프로세스 구조 명확하게 진행
- 서비스 사용자 페르소나 적용 : 실제 서비스 이용 흐름을 고려한 UI/UX 설계
